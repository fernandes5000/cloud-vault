# CloudVault V1

## 1. Resumo executivo

CloudVault V1 e uma plataforma de cloud pessoal / NAS self-hosted, com base monolitica modular e API-first, pronta para evoluir para SaaS multi-tenant sem precisar virar microservicos cedo demais.

Implementado neste workspace:

- backend Laravel 13 com autenticao via Sanctum
- modelo unificado de arquivos e pastas
- upload chunked real
- listagem, favoritos, recentes, lixeira
- links publicos e privados de compartilhamento
- auditoria e notificacoes in-app
- web app Vue 3 + TypeScript + Tailwind integrado a API
- app Flutter com login, i18n e browsing inicial do drive
- Docker, compose e pipeline CI

## 2. Escopo do V1

- contas, login, logout, reset de senha
- verficacao de email por link assinado
- gerenciamento de sessoes por token
- estrutura de quotas
- upload de arquivos em chunks
- pastas, favoritos, recentes e lixeira
- previews seguros para imagens, PDF e video
- links publicos e privados
- notificacoes in-app
- dashboard admin inicial
- web, mobile e estrutura para deploy self-hosted

## 3. Fora do V1

- edicao colaborativa de documentos
- office suite
- sync desktop estilo Dropbox
- colaboracao em tempo real complexa
- OCR, busca full-text e deduplicacao binaria profunda

Justificativa:

- essas areas aumentam muito a superficie operacional e de produto
- nao melhoram o time-to-value do V1 principal
- merecem ciclos separados por causa de storage, locking e UX

## 4. Arquitetura geral

- estilo: monolito modular com API-first
- backend: Laravel 13
- banco recomendado: PostgreSQL
- cache e filas: Redis
- frontend web: Vue 3 + Vite + TypeScript + Tailwind
- mobile: Flutter
- storage: local ou S3-compatible
- deploy: containers com Nginx na borda

Fluxo principal:

1. cliente autentica e recebe token Sanctum
2. cliente cria sessao de upload
3. cliente envia chunks
4. backend monta arquivo, grava no storage final e persiste metadados
5. jobs calculam preview/notificacao
6. web/mobile consomem listagens e links de share

## 5. Backend

Escolhas:

- Laravel 13 pela maturidade do ecossistema, velocidade de entrega e boa ergonomia para auth, filas, policies e storage
- Sanctum para tokens pessoais, dispositivos mobile e SPA sem complexidade de OAuth cedo demais
- Services + FormRequests + Policies + DTOs para manter controllers finos

Modulos:

- `Auth`
- `Drive`
- `Uploads`
- `Shares`
- `Notifications`
- `Admin`
- `Audit`

## 6. Banco de dados

Recomendacao: PostgreSQL em producao.

Justificativa:

- melhor suporte a JSONB e evolucao de metadados
- semantica robusta para constraints e consultas analiticas
- excelente comportamento sob concorrencia
- bom caminho para multi-tenant no futuro

Desenvolvimento local:

- SQLite para testes rapidos
- PostgreSQL no compose para fidelidade de ambiente

## 22. Estrategia de upload

- cliente inicia sessao com `name`, `folder_id`, `total_chunks` e `mime_type`
- cliente envia chunks multipart
- backend grava chunks em area temporaria local
- backend monta arquivo final
- backend grava em disco final configurado
- backend cria `DriveItem` e atualiza quota

## 23. Estrategia de backup mobile

V1 recomendado:

- fila local persistente
- politicas: apenas Wi-Fi, carregando, limite de bateria
- hash e heuristica de duplicacao por nome+tamanho+checksum quando disponivel
- retry com backoff exponencial
- retomar por sessao de upload

No codigo atual:

- ha servico de estrategia e estrutura de app para evoluir essa fila
